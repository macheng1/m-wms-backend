# 后端 PRD 分析

## 使用目标

把产品需求转换成后端可开发的技术任务，输出模块边界、数据模型、接口清单、业务规则、风险点和验收标准。

## 工作流程

1. 先读 PRD、接口文档、现有模块和相近业务代码。
2. 梳理业务对象、核心流程、状态流转和异常分支。
3. 判断是否涉及多租户、权限、库存一致性、订单状态、上传导入、通知、外部服务。
4. 拆出后端交付项：
   - 数据表/实体
   - `dbsql/` SQL 文件，包括建表、字段增删改、初始化数据、维护数据和必要回滚 SQL
   - API 接口
   - DTO 校验
   - Service 业务逻辑
   - 权限与租户隔离
   - 事务和一致性
   - 测试与验证
5. 如果需求缺少关键信息，列出阻塞问题；非阻塞问题给出合理默认假设。

## 输出建议

- 业务目标：一句话说明要解决什么问题。
- 后端范围：哪些模块要改，哪些模块不动。
- 数据设计草案：涉及哪些实体和字段。
- DB SQL 规划：哪些 SQL 需要放入 `dbsql/`，包括 schema、data、rollback。
- 接口清单：路径、方法、入参、出参、权限、租户来源。
- 业务规则：状态、校验、边界、异常。
- 风险点：数据迁移、兼容性、并发、权限、跨租户。
- 验收标准：后端如何证明功能完成。

## 审查重点

- PRD 是否只描述页面，没有描述真实业务规则。
- 是否存在隐含状态机，例如订单、库存、审批、启停用。
- 是否需要审计日志或操作流水。
- 是否有批量导入、重复提交、并发扣减、幂等要求。
- 是否影响已有 API 兼容性。
