# 后端架构治理

## 使用目标

统一项目后端架构和代码组织方式，让后续需求开发按同一套模块边界、分层职责、公共能力和工程规范推进。

## 适用场景

- 用户要求“统一架构”“整理结构”“优化架构”“重构公共逻辑”“规范项目”。
- 多个模块出现重复实现，需要抽公共能力。
- Controller、Service、DTO、Entity 职责混乱。
- 异常、响应、日志、权限、多租户、配置、数据库访问方式不统一。
- 新业务开发前，需要先确定基础架构模式。
- 旧代码需要分阶段治理，避免一次性大改。

## 工作流程

1. 先盘点现状：
   - 目录结构
   - 模块边界
   - 公共能力
   - 异常/响应/日志
   - 鉴权/权限/多租户
   - 数据库访问和事务
   - DTO、Entity、Service、Controller 写法
2. 找出不一致点和风险，不急着改代码。
3. 设计目标架构，明确哪些保留、哪些统一、哪些迁移。
4. 拆成小步改造计划，每一步都有清晰范围和验证方式。
5. 只在用户明确要求实现时改代码；否则先输出方案。
6. 实施时避免一次性大重构，优先做低风险、可验证的局部统一。

## 架构统一范围

- 模块结构：`module`、`controller`、`service`、`dto`、`entities`、`interfaces`、`constants` 的放置规则。
- 分层职责：Controller 只做路由适配，Service 做业务，DTO 做校验，Entity 做表结构。
- 公共能力：异常、响应、分页、导入、上传、日志、权限、多租户、单位换算等。
- 配置管理：环境变量、配置模块、默认值、敏感信息处理。
- 数据库访问：Repository、QueryBuilder、事务、软删除、migration、`dbsql/`。
- API 风格：路径命名、分页返回、错误信息、Swagger、公开接口。
- 安全边界：JWT、RBAC、平台管理员、租户管理员、跨租户隔离。

## 设计原则

- 先统一规则，再迁移代码。
- 不为了抽象而抽象；只有重复、风险或维护成本明显时才抽公共层。
- 不破坏已有 API 兼容性，除非用户明确接受。
- 不把业务差异强行塞进通用抽象。
- 每次重构都要有验证方式。
- 数据库结构调整必须同步 `dbsql/` 和 migration 评估。

## 输出建议

- 当前架构现状。
- 不一致点和风险。
- 目标架构规则。
- 分阶段改造计划。
- 每阶段涉及文件和验证方式。
- 是否需要更新 `AGENTS.md`、`.codex/AGENTS.md` 或相关 skill。

## Review 清单

- 这次改动是否真的改善一致性？
- 是否影响已有接口和前端联调？
- 是否引入跨模块循环依赖？
- 是否破坏多租户、权限或事务边界？
- 是否需要同步文档、SQL、测试或脚本？
