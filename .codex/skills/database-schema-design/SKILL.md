# 数据库表结构设计

## 使用目标

把业务需求转换成稳定、可迁移、可查询、可维护的 MySQL/TypeORM 数据结构；同时负责识别系统运行所需的初始化数据和维护数据，并把人工 SQL 统一沉淀到项目根目录 `dbsql/`。

## 工作流程

1. 从 PRD 或业务流程中识别实体、属性、关系和生命周期。
2. 判断每张表是否属于租户数据，租户表默认包含 `tenantId`。
3. 设计字段类型、长度、是否 nullable、默认值、注释和枚举含义。
4. 设计索引、唯一约束和外键/逻辑关联。
5. 评估是否需要软删除、审计字段、操作流水、历史表。
6. 识别系统必须初始化的数据，例如权限、角色模板、字典、菜单、基础单位、平台默认配置、系统租户、默认管理员或业务默认枚举。
7. 设计建表、新增字段、删除字段、修改字段、索引、约束、初始化数据和维护数据 SQL。
8. 将 SQL 文件统一放入 `dbsql/`，并输出 TypeORM Entity 草案和 migration 影响。

## 设计规则

- 租户业务表优先继承 `TenantBaseEntity`。
- 平台级表和租户级表要明确区分。
- 租户内唯一值使用组合唯一约束，例如 `tenantId + code`。
- 高频查询字段要评估索引：`tenantId`、业务编码、状态、外键、创建时间。
- 金额、数量、库存字段谨慎使用 decimal，并明确精度。
- JSON 字段只用于动态结构，稳定字段应拆成明确列。
- 状态字段要定义清晰枚举，不使用无说明的数字。
- 修改已有实体时必须评估 migration 和历史数据兼容。
- 新增字段、删除字段、修改字段、建表、索引、约束和初始化/维护数据，都要同步产出 `dbsql/` 下的 SQL 文件。
- 删除字段、删除数据、修改字段类型等高风险 SQL 必须写明风险，并提供回滚方案。
- 初始化数据属于 DB 设计职责，不应散落在临时代码、手动操作说明或接口调用步骤中。
- 初始化/维护数据要区分平台级和租户级；租户级数据必须明确 `tenantId` 来源。
- 权限、角色模板、字典、菜单、基础单位、默认配置等数据 SQL 要尽量幂等，避免重复插入。

## 初始化数据职责

DB 设计阶段必须判断是否需要以下数据：

- 权限数据：菜单、按钮、API 权限码。
- 角色数据：平台角色、租户默认角色、角色模板。
- 字典数据：业务状态、类型、分类、枚举映射。
- 菜单数据：前端菜单、路由、排序、父子关系。
- 基础单位：计数、重量、长度、体积等默认单位。
- 平台默认配置：系统参数、门户配置、通知配置。
- 系统基础账号或租户：仅在需求明确需要时设计，并注意安全风险。
- 业务默认数据：新功能上线必须存在的默认分类、默认状态或默认规则。

## dbsql 目录规范

- `dbsql/init-schema.sql`：当前项目初始化建表 SQL。
- `dbsql/YYYYMMDDHHMM_业务说明_schema.sql`：结构变更 SQL，例如建表、新增字段、删除字段、修改字段、索引和约束。
- `dbsql/YYYYMMDDHHMM_业务说明_data.sql`：初始化数据或维护数据 SQL，例如权限、角色模板、字典、菜单、基础单位、平台默认配置。
- `dbsql/YYYYMMDDHHMM_业务说明_rollback.sql`：高风险变更回滚 SQL。
- SQL 文件头部应写明用途、来源需求、影响范围、执行环境和是否可重复执行。
- 数据 SQL 尽量使用幂等写法，避免重复插入或误覆盖。

## 输出建议

- 表清单和职责。
- 字段清单：类型、是否必填、默认值、说明。
- 索引/唯一约束。
- 实体关系。
- `dbsql/` SQL 文件清单和核心 SQL。
- 初始化/维护数据清单：数据用途、归属范围、幂等策略、是否可回滚。
- TypeORM Entity 实现要点。
- migration 方案。
- 风险和回滚方案。

## 常见风险

- 只按页面设计表，忽略业务唯一性和生命周期。
- 忘记 `tenantId` 导致跨租户数据风险。
- 把可查询字段塞进 JSON，后期无法高效筛选。
- 修改字段类型却没有迁移历史数据。
- 状态字段没有状态流转规则。
