# API 契约设计

## 使用目标

在开发前明确接口形态，让前端、后端和测试对路径、参数、返回、权限和错误处理有一致预期。

## 工作流程

1. 先确认业务动作：新增、更新、详情、分页、删除、状态切换、导入、导出、公开接口。
2. 沿用项目已有路由习惯，例如 `page`、`save`、`update`、`detail`、`delete`、`status`。
3. 明确租户来源：登录态、公开入参、平台上下文。
4. 设计 DTO 校验规则和 Swagger 文档。
5. 设计返回结构，列表优先 `{ list, total, page, pageSize }`。
6. 明确错误场景和业务异常信息。

## 设计规则

- 普通业务接口不允许客户端传可信 `tenantId`。
- 公开接口必须显式标注，并校验 `tenantId`、`id` 等关键字段。
- 新增/更新接口要避免客户端写入服务端字段，如 `createdAt`、`updatedAt`。
- 分页查询必须包含页码、页大小、筛选项和稳定排序。
- 文件下载、导入上传等接口要明确 Content-Type 和错误返回。
- 返回给前端的数据不应包含密码、密钥、Token 等敏感字段。

## 输出建议

- 接口列表：方法、路径、说明。
- 入参 DTO：字段、类型、是否必填、校验规则。
- 返回示例。
- 错误场景。
- 权限要求。
- 租户来源。
- Swagger 更新点。

## Review 清单

- 前端是否能根据契约直接联调？
- 是否破坏已有 API？
- 是否有跨租户参数风险？
- 是否覆盖异常分支？
- DTO 校验是否足够表达业务边界？
